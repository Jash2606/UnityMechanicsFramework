This document provides a line-by-line explanation of the Save & Load System.
It covers the architecture, every file's purpose, and the reasoning behind
each design decision.

================================================================================
  FILE 1: ISaveable_UMFOSS.cs — The Interface Contract
================================================================================

public interface ISaveable_UMFOSS
// Defines the contract that ANY script must implement to participate in saving.
// This is the key to the entire architecture: the SaveSystem never needs to know
// what HealthSystem or InventorySystem are. It only knows ISaveable_UMFOSS.

    string GetSaveID();
    // Returns a unique string identifier for this saveable instance.
    // This ID is the key used to store and retrieve the state in the save file.
    //
    // WHY A SERIALIZED FIELD (uniqueID) INSTEAD OF gameObject.name:
    //   - gameObject.name can change at runtime (e.g., "Player(Clone)")
    //   - Multiple instances might share the same gameObject.name
    //   - Dynamically spawned objects have unpredictable names
    //   - A serialized uniqueID field is stable, inspector-editable, and predictable
    //
    // RULE: Two saveables with the same ID = last one overwrites the other.
    //       Always ensure IDs are unique.

    object CaptureState();
    // Called by SaveSystem when saving.
    // The implementing script packs its current state into a serializable object
    // (typically a [System.Serializable] struct) and returns it.
    //
    // WHY EACH MECHANIC OWNS ITS OWN DATA FORMAT:
    //   A monolithic SaveManager that knows about HealthSystem, InventorySystem,
    //   and WeaponSystem is coupled to all three. Add a new mechanic — modify
    //   SaveManager. Rename a field — modify SaveManager.
    //   The ISaveable pattern inverts this. Each mechanic knows everything about
    //   its own data. SaveSystem knows nothing about any mechanic.

    void RestoreState(object state);
    // Called by SaveSystem when loading.
    // The state parameter is a JSON string that was previously captured.
    // The implementing script deserializes it back to its struct type
    // using JsonUtility.FromJson<T>() and applies the values.

================================================================================
  FILE 2: SaveData_UMFOSS.cs — The Save File Structure
================================================================================

public class SaveData_UMFOSS
// This is the complete save file as it exists in memory.
// A single instance of this class = one save file on disk.

    public int saveVersion;
    // The version of the save format when this file was created.
    // WHY VERSION:
    //   Games change after release. New fields are added, mechanics are renamed.
    //   Without versioning, old save files become invalid after an update.
    //   With versioning, the system can detect outdated saves and migrate them.

    public string saveSlotName;
    // Which slot this save belongs to (e.g., "Slot1", "Slot2", "AutoSave").

    public string lastSavedTimestamp;
    // ISO 8601 timestamp (e.g., "2024-01-15T14:30:00.000Z").
    // Used by the save slot UI to show when each slot was last saved.

    public string sceneNameOnSave;
    // The active scene when the save was made.
    // WHY STORE THE SCENE:
    //   Load must know which scene to load BEFORE calling RestoreState.
    //   The saveables don't exist until their scene is loaded.
    //   Without this, loading always starts from the same scene regardless
    //   of where the player saved.

    public SerializableDictionary savedObjects;
    // Key = SaveID (from GetSaveID())
    // Value = JSON string (from JsonUtility.ToJson(CaptureState()))
    //
    // Each saveable's state is independently serialized to its own JSON string.
    // This means saveables are completely independent — they can be added,
    // removed, or modified without affecting each other.

public class SerializableDictionary : ISerializationCallbackReceiver
// WHY THIS EXISTS:
//   Unity's JsonUtility cannot serialize Dictionary<TKey, TValue> natively.
//   It throws a silent error and produces empty JSON.
//
//   ISerializationCallbackReceiver solves this:
//   - OnBeforeSerialize: converts Dictionary → two parallel Lists (keys + values)
//   - OnAfterDeserialize: converts two Lists → Dictionary
//
//   This avoids adding Newtonsoft.Json as a package dependency.

================================================================================
  FILE 3: SaveFileHandler_UMFOSS.cs — File I/O
================================================================================

public static class SaveFileHandler_UMFOSS
// A static utility class. No MonoBehaviour, no scene dependency.
// It ONLY knows how to read and write files.
// The SaveSystem calls it. Nothing else should.

    SaveDirectory → Application.persistentDataPath + "/Saves/"
    // WHY persistentDataPath AND NOT dataPath:
    //   - dataPath is the project's Assets folder — READONLY on builds
    //   - persistentDataPath is the OS-designated save location:
    //       Windows: C:/Users/<user>/AppData/LocalLow/<company>/<game>/
    //       macOS:   ~/Library/Application Support/<company>/<game>/
    //       Linux:   ~/.config/unity3d/<company>/<game>/
    //       Android: /data/data/<package>/files/
    //       iOS:     /var/mobile/Containers/Data/Application/<guid>/Documents/
    //   - It persists across app updates
    //   - It's the correct cross-platform save location

    Save():
    // 1. Ensure directory exists (Directory.CreateDirectory is idempotent)
    // 2. Serialize SaveData to JSON with JsonUtility.ToJson
    // 3. Optionally XOR-encrypt the JSON string
    // 4. Write to disk with File.WriteAllText
    // All wrapped in try-catch for IO errors, permission issues, disk full, etc.

    Load():
    // 1. Check file exists — return null if not (no exception)
    // 2. Read file contents
    // 3. Optionally XOR-decrypt
    // 4. Deserialize with JsonUtility.FromJson<SaveData_UMFOSS>
    // 5. Return null on any failure — NEVER throw. Caller handles null gracefully.

    XOREncryptDecrypt():
    // Simple symmetric XOR cipher.
    // NOT cryptographically strong. This is anti-tamper for casual games.
    // Keeps honest players honest without adding complexity.
    // The same function encrypts and decrypts (XOR is its own inverse).

================================================================================
  FILE 4: SaveSystem_UMFOSS.cs — The Coordinator
================================================================================

public class SaveSystem_UMFOSS : MonoSingletongeneric<SaveSystem_UMFOSS>
// Inherits from the singleton base so it:
//   - Persists across scene loads (DontDestroyOnLoad)
//   - Is globally accessible via SaveSystem_UMFOSS.Instance
//   - Automatically destroys duplicates

    Register() / Deregister():
    // ISaveable scripts call these in OnEnable/OnDisable (not Start/OnDestroy).
    //
    // WHY OnEnable/OnDisable:
    //   OnEnable and OnDisable fire when an object is activated/deactivated
    //   in the hierarchy — including during scene load and unload.
    //   Start and OnDestroy do NOT fire reliably during scene transitions.
    //   Using the correct lifecycle methods means saveables are always
    //   correctly registered when a save is triggered.
    //
    // NULL-SAFETY: Register() checks for null saveable, null/empty SaveID,
    // and handles duplicate IDs with a warning (overwrites, doesn't crash).

    Save():
    // 1. Create a new SaveData_UMFOSS
    // 2. For each registered ISaveable:
    //    a. Call CaptureState() — get the state object
    //    b. Serialize it to JSON string with JsonUtility.ToJson()
    //    c. Store in savedObjects under GetSaveID() key
    // 3. Set metadata (timestamp, scene, version)
    // 4. Pass to SaveFileHandler.Save()
    // 5. Publish OnGameSaved event on success / OnSaveFailed on failure

    Load():
    // 1. Call SaveFileHandler.Load() — get SaveData
    // 2. Check saveVersion:
    //    - If newer than current → BLOCK. Log error + OnLoadFailed.
    //      (Player running older build with a save from newer build.)
    //    - If older → call MigrateSave() to upgrade
    // 3. Check scene: if different from current, load scene async first
    // 4. For each registered ISaveable:
    //    a. Look up GetSaveID() in SaveData.savedObjects
    //    b. If found: pass JSON string to RestoreState()
    //    c. If not found: skip gracefully (new mechanic added after save)
    // 5. Publish OnGameLoaded

    MigrateSave():
    // Switch on the old version, apply cumulative upgrades.
    // Each case falls through to the next using goto.
    // This is the standard version migration pattern.
    // Future game updates add new cases here.

    pendingLoadData:
    // When loading across scenes, the save data is cached here.
    // As new saveables register (via OnEnable after scene loads),
    // they check for pending data and restore immediately.
    // This solves the timing problem: saveables don't exist until
    // their scene is loaded, but they need data from the save file.

================================================================================
  FILE 5: SaveSystemEvents_UMFOSS.cs — Events
================================================================================

    Delegate definitions for all save system events.
    Kept in a separate file for clean organization.
    Events include: OnGameSaved, OnGameLoaded, OnSaveDeleted,
    OnSaveFailed, OnLoadFailed, OnNewGameStarted.

    WHY EVENTS:
    Any system (UI, analytics, audio) can react to save/load operations
    without the SaveSystem needing to know they exist.
    This follows the Observer pattern used throughout the framework.

================================================================================
  ARCHITECTURAL NOTES
================================================================================

WHY NOT PlayerPrefs:
  PlayerPrefs stores key-value pairs of string, int, and float.
  It works for settings (volume, resolution). It breaks for anything
  structured: an inventory with 20 slots, a quest log with completion states,
  a map with discovered areas. No save slots, no versioning, no file you can
  inspect or back up. The moment a game has real progression, PlayerPrefs
  is the wrong tool.

WHY persistentDataPath:
  See SaveFileHandler section above. It's the ONLY correct cross-platform
  save location. dataPath is readonly in builds. StreamingAssets is readonly.
  Application.temporaryCachePath can be cleared by the OS.

WHY ISaveable PATTERN (Dependency Inversion):
  Without ISaveable: SaveManager.Save() has 40+ lines of hardcoded
  "PlayerPrefs.SetFloat("health", player.health)". Every new mechanic
  requires modifying SaveManager.

  With ISaveable: Each mechanic handles its own data. SaveSystem calls
  three interface methods. Zero changes to SaveSystem when adding mechanics.
  This is the Open/Closed Principle in practice.

WHY STABLE UNIQUE IDs:
  Using gameObject.name is fragile:
  - "Player" might become "Player(Clone)" when instantiated
  - Two NPCs named "Guard" would overwrite each other's data
  - Scene reload can change object names unpredictably
  
  A [SerializeField] string uniqueID is:
  - Set once in the Inspector
  - Stable across scene loads, instantiation, and builds
  - Visible and editable by designers
  - Deterministic — same ID always maps to the same saveable

================================================================================
  HOW TO ADD A NEW SAVEABLE TO YOUR MECHANIC
================================================================================

1. Add ": ISaveable_UMFOSS" to your class declaration
2. Add [SerializeField] private string uniqueID = "MyMechanic_Player";
3. Create a [System.Serializable] struct with your save data fields
4. Implement GetSaveID() → return uniqueID;
5. Implement CaptureState() → return new YourSaveData { ... };
6. Implement RestoreState(object state) →
       string json = state as string;
       YourSaveData data = JsonUtility.FromJson<YourSaveData>(json);
       // Apply data to your fields
7. In OnEnable: SaveSystem_UMFOSS.Instance?.Register(this);
8. In OnDisable: SaveSystem_UMFOSS.Instance?.Deregister(this);

That's it. Zero changes to SaveSystem. Your mechanic is now saved and loaded.
